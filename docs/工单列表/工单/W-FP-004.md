## 工单 ID

W-FP-004

## 工单标题

修复悬浮窗模式无法单独使用、无法稳定显示弹幕与停止后残留的问题

## 背景

当前仓库已完成 W-FP-001/002/003，悬浮窗字段、组件本体与 Web 设置项均已接入；但实际排查发现 3 个具体问题：

1. `display_mode="floating_panel"` 时，`DanmuApp.start()` 与 `_on_config_changed()` 仍无条件启动/显示横向 Overlay，导致“仅悬浮窗”语义未真正落地。
2. 悬浮窗分发挂在 `_consume_reply_queue()` 的 `if item:` 分支内；当 `DanmuEngine.add_text()` 因去重、入口区过载或轨道/布局拒收返回 `None` 时，悬浮窗也收不到这条弹幕，导致“悬浮窗模式不出弹幕”。
3. `DanmuApp.stop()` 只隐藏 Overlay，不处理 `floating_panel`；暂停/停止后可能残留悬浮窗可见状态或旧条目。

本工单只修复上述 3 个已确认问题，不扩展悬浮窗功能。

## 目标

完成后：

- 选择“仅悬浮窗”时，横向 Overlay 不再显示；
- 选择“仅悬浮窗”或“两者并存”时，只要回复队列成功消费到一条有效弹幕文本，悬浮窗都能收到并显示，不再依赖 `DanmuEngine.add_text()` 是否成功；
- 点击停止/暂停后，悬浮窗立即隐藏且不保留旧条目；再次开始时按当前 `display_mode` 干净恢复；
- `display_mode=overlay / floating_panel / both` 三种模式语义与文档一致。

## 需求

1. **修正“仅悬浮窗”模式语义**：
   - `display_mode="floating_panel"` 时，不启动/不显示横向 Overlay；
   - `display_mode="both"` 时，Overlay 与悬浮窗同时显示；
   - `display_mode="overlay"` 时，仅显示横向 Overlay；
   - 启动路径与运行中配置切换路径行为一致，不允许“启动正确、切换错误”或“切换正确、重启错误”。

2. **悬浮窗分发改为独立于 `DanmuEngine.add_text()` 成功与否**：
   - `_consume_reply_queue()` 中，悬浮窗是否接收弹幕只取决于：
     - 当前 reply 是否有有效 `display_content`；
     - 当前 `display_mode` 是否启用悬浮窗；
   - 不再要求 `item is not None` 才能分发给悬浮窗；
   - 仍然保持主链路稳健性：悬浮窗异常只记 debug，不阻断主链路。

3. **补齐 stop/start 生命周期**：
   - `stop()` 后悬浮窗必须立即隐藏；
   - `stop()` 后悬浮窗内部旧条目必须清空，避免再次 `start()` 时旧弹幕继续滚动；
   - 再次 `start()` 时根据当前 `display_mode` 重新显示/保持隐藏，行为稳定。

4. **如需补公开方法，保持最小 API 面**：
   - 若 `main.py` 无法在不访问私有字段的前提下完成 stop 清理，可在 `app/floating_panel.py` 增加单一职责的公开方法，例如“隐藏并清空运行态”；
   - 禁止顺手重构悬浮窗绘制、拖拽、Win32 样式、字号/速度等既有实现。

5. **回归测试必须覆盖这 3 个问题**：
   - `display_mode="floating_panel"` 时 Overlay 不显示；
   - `DanmuEngine.add_text()` 返回 `None` 时，悬浮窗仍能收到弹幕；
   - `stop()` 会隐藏并清空悬浮窗，`start()` 后不会恢复旧条目；
   - `display_mode="both"` 时，两条显示链路都工作；
   - 运行中通过配置切换 `display_mode` 时，三种模式行为正确。

## 手动验证步骤

1. 运行 `python main.py`。
2. Web 控制台 → 助手设置 → 「弹幕显示」tab，选择“仅悬浮窗”并保存。
3. 观察 5 秒内横向 Overlay 不出现，仅悬浮窗显示弹幕。
4. 切换为“两者并存”并保存，确认横向 Overlay 与悬浮窗同时显示。
5. 触发一组容易被横向去重/拒收的重复弹幕，确认悬浮窗仍持续出弹。
6. 点击停止/暂停，确认悬浮窗立即隐藏；再次开始后不继续滚动停止前的旧条目。

## 风险点

- 本工单触达 `main.py::_consume_reply_queue()`、`start()`、`stop()`，属于主链路高敏感区；必须做最小改动，禁止重排主链路顺序。
- 若为 stop 清理给 `FloatingPanel` 增加公开方法，方法职责必须单一，避免把生命周期逻辑散落到多个入口。
- “悬浮窗不依赖 Overlay 上屏成功”不等于重发 AI；仍然只能消费现有 reply queue 的文本结果。
